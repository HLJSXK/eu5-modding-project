EU5 MOD STRUCTURE ANALYSIS - SUMMARY
=====================================
Based on analysis of 16 real EU5 mods from Steam Workshop

TOTAL MODS ANALYZED: 16
- Gameplay mods: 10
- Translation mods: 3
- UI/Graphics mods: 3

KEY FINDINGS:
=============

1. MANDATORY STRUCTURE
----------------------
Every mod MUST have:
  /.metadata/
    └── metadata.json          (Required: defines mod identity)
  
Optional top-level folders:
  /in_game/                    (Game content modifications)
  /main_menu/                  (Localization, graphics)
  /loading_screen/             (Loading screen customization)
  /script/                     (Custom scripts - rare)

2. METADATA.JSON STRUCTURE
--------------------------
Required fields:
  - name: Display name
  - id: Unique identifier (format: author.modname or empty)
  - version: Mod version
  - supported_game_version: Compatible game version (e.g., "1.0.7" or "1.*.*")
  - short_description: Brief description (supports BBCode for Steam)
  - tags: Array of category tags
  - relationships: Mod dependencies (usually empty array)
  - game_custom_data: Game-specific settings (e.g., multiplayer_synchronized)

Optional fields:
  - picture/thumbnail: Preview image filename

4. MAIN_MENU FOLDER STRUCTURE
------------------------------
/main_menu/localization/      (Localization files)
  └── *_l_english.yml         (English localization)
  └── *_l_simp_chinese.yml    (Simplified Chinese)
  └── [other languages]

/main_menu/gfx/               (Graphics assets)
  └── interface/              (UI graphics)
      ├── illustrations/      (Event illustrations)
      └── [other graphics]

5. FILE FORMATS
---------------
- .json: Metadata, configuration
- .txt: Game scripts (events, common definitions)
- .yml: Localization (MUST be UTF-8-BOM encoded)
- .gui: GUI definitions
- .dds: Texture files (DirectDraw Surface)
- .png: Images (thumbnails, icons)

6. COMMON PATTERNS
------------------
a) Localization files:
   - Format: *_l_<language>.yml
   - Must start with UTF-8 BOM
   - Structure: l_english:\n key: "value"

b) Event files:
   - Namespace declaration at top
   - Event IDs: namespace.number
   - Must be explicitly triggered (no MTTH)

c) GUI files:
   - Type definitions
   - Widget hierarchies
   - Datacontext bindings

d) Common scripts:
   - Key-value pairs
   - Nested blocks with braces
   - Trigger and effect blocks

7. MOD CATEGORIES (by complexity)
----------------------------------
Simple mods (1-20 files):
  - Balance tweaks
  - Small UI changes
  - Single feature additions

Medium mods (20-100 files):
  - Content expansions
  - Multiple features
  - Localization packages

Complex mods (100+ files):
  - Total conversions
  - Major overhauls
  - Comprehensive content packs

8. BEST PRACTICES OBSERVED
---------------------------
- Use descriptive file names with prefixes (e.g., btc_, htc_)
- Organize by feature/namespace
- Include thumbnail.png for workshop
- Use proper versioning
- Specify supported game version
- Include localization for all text
- Use UTF-8-BOM for .yml files
- Avoid overwriting vanilla files when possible
- Use numbered prefixes for load order (e.g., 00_, 01_)

9. COMMON MOD TYPES
-------------------
- Gameplay: New mechanics, balance changes
- Historical: Historical events, missions
- UI: Interface improvements
- Graphics: Visual enhancements
- Translation: Localization
- Fixes: Bug fixes, improvements
- Map: Map modifications
- Military: Unit types, combat
- Diplomacy: Diplomatic options
- Economy: Trade, production

10. MULTIPLAYER CONSIDERATIONS
-------------------------------
- Set "multiplayer_synchronized": true in game_custom_data
- Ensure all players have same mod version
- Test for desync issues
- Avoid client-side only modifications
